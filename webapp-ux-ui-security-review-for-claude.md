# ข้อเสนอแนะ UX/UI, Accessibility, Security และ Privacy สำหรับ Fun English Journey

อัปเดต: 11 กรกฎาคม 2026  
วัตถุประสงค์: เอกสารส่งต่องานให้ Claude สำหรับปรับปรุงเว็บแอปให้ใช้งานได้ดีและปลอดภัยบนมือถือ แท็บเล็ต และคอมพิวเตอร์

## สรุปภาพรวม

เว็บแอปมีฐานการออกแบบสำหรับมือถือที่ดี ปุ่มหลักมีขนาดเหมาะกับการสัมผัส สีสันและรูปแบบสอดคล้องกับกลุ่มเด็ก และหน้าเริ่มต้นไม่เกิด horizontal overflow บนหน้าจอมือถือขนาด 390 × 844 px

อย่างไรก็ตาม ก่อนเผยแพร่ใช้งานจริง ควรแก้ประเด็นสำคัญด้าน XSS, ความเป็นส่วนตัวของเสียงเด็ก, Accessibility, Responsive Navigation และการบันทึกความคืบหน้าระหว่างบทเรียน

## Priority 1 — ต้องแก้ก่อนเผยแพร่

### 1. ป้องกัน Stored DOM XSS จากชื่อผู้เล่น

ตำแหน่งหลัก: `fun-english-journey/js/engine.js` บริเวณ `refreshProfilesList()`

ชื่อผู้เล่นถูกอ่านจาก IndexedDB แล้วนำไปประกอบใน `innerHTML` โดยตรง:

- `p.name`
- ข้อมูลอื่นจาก profile ที่ถูกแทรกใน HTML template

แม้ช่องชื่อมี `maxlength="12"` แต่ไม่ได้ป้องกัน HTML injection

**สิ่งที่ควรทำ**

- สร้าง DOM element ด้วย `document.createElement()`
- กำหนดข้อมูลผู้ใช้ด้วย `textContent`
- ห้ามนำค่าจากผู้ใช้เข้า `innerHTML`
- ตรวจชื่อด้วย allowlist เช่น ตัวอักษรไทย อังกฤษ ตัวเลข ช่องว่าง และอักขระที่อนุญาต
- อย่าพึ่งพา client-side validation เพียงอย่างเดียว หากอนาคตมี backend

### 2. ลดการใช้ `innerHTML` กับข้อมูลบทเรียน

ข้อมูลจาก JSON เช่น ชื่อบท คำศัพท์ คำถาม และตัวเลือก ถูกนำไปสร้าง HTML หลายตำแหน่ง

ปัจจุบัน JSON เป็นไฟล์ภายในโครงการ ความเสี่ยงจึงต่ำกว่าข้อมูลผู้ใช้ แต่หากอนาคตมี CMS, ระบบหลังบ้าน, ครูอัปโหลดบทเรียน หรือดาวน์โหลดเนื้อหาจากภายนอก จะกลายเป็นช่องทาง Stored XSS

**สิ่งที่ควรทำ**

- ข้อความทั่วไปใช้ `textContent`
- URL รูปภาพและเสียงต้องผ่านการตรวจสอบรูปแบบและ origin
- หากจำเป็นต้องรองรับ HTML ให้ใช้ sanitizer ที่เชื่อถือได้และกำหนด allowlist ที่จำกัด
- สร้าง helper สำหรับ render ข้อความอย่างปลอดภัย เพื่อไม่ให้แต่ละหน้าจอใช้วิธีต่างกัน

### 3. เพิ่ม Privacy Flow ก่อนขอใช้ไมโครโฟน

แอปใช้:

- `navigator.mediaDevices.getUserMedia({audio:true})`
- `MediaRecorder`
- `SpeechRecognition` หรือ `webkitSpeechRecognition`

ไฟล์เสียงที่อัดถูกเก็บเป็น Blob ชั่วคราว แต่ Speech Recognition ของบางเบราว์เซอร์อาจประมวลผลผ่านบริการภายนอกของผู้ผลิตเบราว์เซอร์

**สิ่งที่ควรทำ**

- ก่อนขอ permission ครั้งแรก แสดงคำอธิบายแบบภาษาง่ายสำหรับเด็กและผู้ปกครอง
- ระบุว่าเสียงใช้ทำอะไร เก็บไว้นานเท่าใด และส่งออกจากอุปกรณ์หรือไม่
- ห้ามรับรองว่า “เสียงไม่ออกจากอุปกรณ์” จนกว่าจะยืนยันสถาปัตยกรรมได้
- ให้ผู้เรียนข้ามกิจกรรมไมโครโฟนได้โดยไม่เสียสิทธิ์เรียนต่อ
- แสดงสถานะขณะไมโครโฟนทำงานอย่างชัดเจน
- ปิด media tracks ทุกครั้งเมื่ออัดเสร็จ ออกจากบท หรือเกิด error
- เรียก `URL.revokeObjectURL()` เมื่อไม่ใช้ไฟล์เสียงชั่วคราวแล้ว
- หากใช้กับเด็กจริง ควรมีข้อความยินยอมสำหรับผู้ปกครองตามรูปแบบการใช้งานและกฎหมายที่เกี่ยวข้อง

### 4. เอา `user-scalable=no` ออก

ตำแหน่ง: `fun-english-journey/index.html`

ปัจจุบัน viewport meta ปิดการซูมบนมือถือ ซึ่งไม่เหมาะกับผู้มีสายตาเลือนรางหรือผู้ที่ต้องการขยายข้อความ

ควรใช้:

```html
<meta name="viewport" content="width=device-width, initial-scale=1.0">
```

### 5. ใช้ HTTPS เท่านั้นใน production

ไมโครโฟนและ Web APIs หลายรายการต้องใช้ secure context โดยทั่วไปจะทำงานบน HTTPS หรือ localhost เท่านั้น

**สิ่งที่ควรทำ**

- บังคับ HTTPS
- redirect HTTP ไป HTTPS
- เปิด HSTS เมื่อระบบพร้อมและตรวจสอบ subdomain แล้ว
- ไม่โหลด asset หรือ API ผ่าน HTTP

## Priority 2 — UX/UI และ Accessibility ที่ควรทำในรอบถัดไป

### 6. ปรับแถบเลือกชั้น ป.1–ป.6 บนมือถือ

ปุ่มชั้นเรียน 6 ปุ่มอยู่ในแถวเดียว เมื่อแสดงบนมือถือจะคับและพื้นที่แตะอาจเล็กเกินไป

ตัวเลือกที่แนะนำ:

- แสดงเป็น 2 แถว แถวละ 3 ชั้น หรือ
- ทำ horizontal scroll โดยปุ่มกว้างอย่างน้อย 72–88 px หรือ
- ให้ผู้ปกครองกำหนดชั้นหลักในโปรไฟล์ แล้วเก็บการเปลี่ยนชั้นไว้ในเมนู

สำหรับเด็ก วิธีที่แนะนำที่สุดคือกำหนด “ชั้นหลัก” ต่อโปรไฟล์ เพื่อลดการเลือกผิดและลดภาระการตัดสินใจ

### 7. ใช้พื้นที่แท็บเล็ตและคอมพิวเตอร์ให้ดีขึ้น

`.screen` ถูกจำกัด `max-width: 520px` จึงอ่านง่าย แต่บนแท็บเล็ตและจอคอมพิวเตอร์มีพื้นที่ว่างมาก และรายการบทเรียนยาวเกินจำเป็น

แนวทาง Responsive Layout:

- มือถือ: 1 คอลัมน์
- แท็บเล็ต: แผนที่หรือการ์ดบทเรียน 2 คอลัมน์
- คอมพิวเตอร์: รายการหน่วยด้านซ้ายและเนื้อหาด้านขวา
- จำกัดความกว้างของข้อความเพื่อให้อ่านง่าย แต่ไม่จำกัด UI ทั้งหมดไว้ที่ 520 px

### 8. บันทึกความคืบหน้าระหว่างบทเรียน

ปัจจุบันบันทึกหลัก ๆ เฉพาะ XP และดาวหลังจบบท หากเด็กปิดแท็บหรือออกกลางบท อาจต้องเริ่มใหม่

ควรบันทึก:

- lesson ID
- step ปัจจุบัน
- XP ที่ได้ในรอบนั้น
- คำตอบหรือรายการที่ทำสำเร็จเท่าที่จำเป็น
- เวลาอัปเดตล่าสุด

เมื่อกลับมา ให้เลือก:

- เรียนต่อจากเดิม
- เริ่มบทใหม่

### 9. แสดงระยะเวลาและจำนวนกิจกรรมก่อนเริ่ม

แต่ละบทอาจมีหลายขั้น ได้แก่ คำศัพท์ ฟัง จับคู่ เรียงคำ ฝึกพูด ควิซ และแปลงประโยค

ควรแสดงก่อนเริ่ม เช่น:

- ใช้เวลาประมาณ 8–12 นาที
- มีกิจกรรมทั้งหมดกี่ขั้น
- สามารถพักแล้วกลับมาเรียนต่อได้

### 10. เพิ่ม Hint และ Feedback ที่ช่วยให้เรียนรู้

ปัจจุบันเมื่อผิดมักแจ้งเพียง “ลองใหม่”

แนะนำ flow:

1. ผิดครั้งแรก: ให้ลองใหม่
2. ผิดครั้งที่สอง: ให้คำใบ้
3. ผิดครั้งที่สาม: แสดงคำตอบพร้อมเสียงหรือคำอธิบายสั้น
4. นำข้อนั้นกลับมาถามอีกครั้งช่วงท้าย

ควรเพิ่มปุ่ม “ขอคำใบ้” หรือ “ยังไม่รู้” เพื่อลดการกดเดาสุ่ม

### 11. ป้องกันการออกจากบทเรียนโดยไม่ตั้งใจ

ปุ่มรูปบ้านออกจากบทได้ทันที เด็กอาจแตะผิด

ควร:

- บันทึกอัตโนมัติก่อนออก
- ใช้ปุ่มที่มี label ชัดเจน
- ถ้าข้อมูลยังไม่บันทึก ให้ถามว่าจะพักบทหรือออก
- หลีกเลี่ยง modal ที่ถามซ้ำทุกครั้ง หากระบบบันทึกอัตโนมัติได้อย่างน่าเชื่อถือ

### 12. เพิ่มเมนูการตั้งค่าเสียงและภาพเคลื่อนไหว

ควรมีตัวเลือก:

- ปิดเสียงเอฟเฟกต์
- ปรับระดับเสียง
- เลือกเสียงช้าหรือความเร็วธรรมชาติ
- ปิดการอ่านภาษาไทยอัตโนมัติ
- ลดภาพเคลื่อนไหว

เพิ่ม CSS สำหรับ `prefers-reduced-motion` เพื่อปิดหรือลด `pop`, `shake`, `pulse` และ `bounce`

## Accessibility Checklist

### 13. เพิ่มชื่อให้ปุ่มอีโมจิ

ปุ่มเช่น `🦁`, `🐼`, `🏠` และ `👥` ควรมี `aria-label`

ตัวอย่าง:

```html
<button aria-label="เลือกตัวละครสิงโต">🦁</button>
<button aria-label="กลับหน้าแผนที่">🏠</button>
```

### 14. เพิ่ม `<label>` ให้ช่องชื่อ

ข้อความที่อยู่ก่อนช่องกรอกไม่ทดแทน `<label for="inp-name">`

ควรเชื่อม label กับ input โดยตรง และเพิ่มคำอธิบายข้อจำกัดชื่อหากจำเป็น

### 15. ประกาศสถานะการเลือกให้ Assistive Technology

- ตัวละคร: ใช้ `aria-pressed="true/false"`
- แท็บระดับชั้น: ใช้ `role="tablist"`, `role="tab"`, `aria-selected`
- Progress bar: ใช้ `role="progressbar"`, `aria-valuemin`, `aria-valuemax`, `aria-valuenow`

### 16. เพิ่ม Live Region สำหรับผลคำตอบ

Feedback เช่น “ถูกต้อง”, “ลองใหม่” หรือคะแนนพูด ควรอยู่ในพื้นที่:

```html
<div aria-live="polite" role="status"></div>
```

ข้อความสำคัญที่ต้องประกาศทันทีอาจใช้ `aria-live="assertive"` อย่างจำกัด

### 17. เพิ่ม Keyboard Focus ที่เห็นชัด

CSS ยังควรเพิ่ม `:focus-visible` สำหรับปุ่ม ลิงก์ input chip และตัวเลือกทั้งหมด

ทดสอบด้วย:

- Tab และ Shift+Tab
- Enter
- Space
- Escape สำหรับปิด dialog

ทุกกิจกรรมต้องทำได้โดยไม่ใช้เมาส์หรือหน้าจอสัมผัส

### 18. อย่าใช้สีเพียงอย่างเดียว

สถานะถูกและผิดใช้เขียวกับแดง ควรมี:

- ไอคอน
- ข้อความ
- รูปทรงหรือ border ที่ต่างกัน
- feedback ทางเสียงที่สามารถปิดได้

### 19. ตรวจ Contrast และขนาดข้อความ

- ตรวจสีข้อความรอง `#6b7a99` บนพื้นหลังแต่ละชนิด
- รักษาข้อความหลักไม่น้อยกว่า 16 px บนมือถือ
- touch target ควรประมาณ 44 × 44 px ขึ้นไป
- อย่าใช้ข้อความขนาดเล็กเป็นข้อมูลสำคัญเพียงตำแหน่งเดียว

## Security Hardening

### 20. ย้าย Inline Event Handler ออกจาก HTML

ปัจจุบันมี `onclick` และ `onerror` แบบ inline หลายตำแหน่ง ทำให้กำหนด Content Security Policy ที่เข้มงวดได้ยาก

ควร:

- ใช้ `addEventListener`
- ใช้ event delegation สำหรับรายการที่ render แบบ dynamic
- ย้าย inline style ไป CSS class
- หลีกเลี่ยงการสร้าง JavaScript code เป็น string

### 21. เพิ่ม Content Security Policy

หลังย้าย inline scripts และ handlers แล้ว ควรกำหนด CSP ที่ยึด `'self'` เป็นหลัก

แนวทางเริ่มต้น ต้องปรับตามระบบ hosting จริง:

```text
default-src 'self';
script-src 'self';
style-src 'self';
img-src 'self' data: blob:;
media-src 'self' blob:;
font-src 'self';
connect-src 'self';
object-src 'none';
base-uri 'none';
frame-ancestors 'none';
form-action 'self';
```

ไม่ควรใช้ `'unsafe-inline'` เป็นคำตอบถาวร

### 22. Self-host ฟอนต์

ปัจจุบันโหลด Google Fonts ซึ่ง:

- ทำให้ใช้งานออฟไลน์ไม่สมบูรณ์
- เพิ่ม dependency ภายนอก
- อาจสร้างประเด็น privacy หรือ availability
- ทำให้ CSP ซับซ้อนขึ้น

ควร self-host ฟอนต์ หรือใช้ system font stack ที่เหมาะสม

### 23. เพิ่ม Security Headers

ตั้งค่าผ่าน hosting/server:

```text
X-Content-Type-Options: nosniff
Referrer-Policy: no-referrer
Permissions-Policy: microphone=(self), camera=(), geolocation=()
Cross-Origin-Opener-Policy: same-origin
```

พิจารณาเพิ่มเติมตามโครงสร้าง deployment:

- `Strict-Transport-Security`
- `Cross-Origin-Resource-Policy`
- `Cross-Origin-Embedder-Policy` เฉพาะเมื่อทดสอบ compatibility แล้ว

### 24. จำกัดและตรวจสอบ Asset URL

- ไม่อนุญาต `javascript:` หรือ URL protocol ที่ไม่จำเป็น
- โหลดรูป เสียง และ JSON จาก same origin
- ตรวจ filename/slug ก่อนประกอบ path
- ตั้ง `X-Content-Type-Options: nosniff`
- กำหนด MIME type ของ JSON, audio, image และ JavaScript ให้ถูกต้อง

### 25. จัดการข้อผิดพลาดโดยไม่เปิดเผยรายละเอียดภายใน

หน้าผู้ใช้ควรเห็นข้อความทั่วไป เช่น “โหลดบทเรียนไม่สำเร็จ กรุณาลองใหม่”

รายละเอียด error, stack trace, URL ภายใน และข้อมูลระบบควรอยู่ใน developer log ที่ไม่มีข้อมูลส่วนตัวของเด็ก

## Data Protection สำหรับข้อมูลเด็ก

### 26. เก็บข้อมูลให้น้อยที่สุด

ปัจจุบันควรเก็บเพียง:

- ชื่อเล่น
- avatar
- ความคืบหน้า
- XP และดาว
- การตั้งค่าที่จำเป็น

ไม่ควรเก็บชื่อจริง โรงเรียน ห้องเรียน วันเกิด อีเมล ตำแหน่ง หรือรหัสประจำตัวเด็ก หากไม่มีความจำเป็นชัดเจน

### 27. เพิ่มการจัดการโปรไฟล์

ควรมี:

- เปลี่ยนชื่อเล่นและ avatar
- ลบโปรไฟล์
- ลบข้อมูลทั้งหมด
- ยืนยันก่อนลบ
- อธิบายว่าผู้ใช้อุปกรณ์เดียวกันอาจเห็นโปรไฟล์ได้

ถ้ามี export/import ในอนาคต ต้องระวังข้อมูลเด็กและตรวจไฟล์นำเข้าก่อนใช้งาน

### 28. แยกสถานะ Local-only กับ Online Account

หากเพิ่มระบบบัญชีออนไลน์ในอนาคต ต้องออกแบบใหม่สำหรับ:

- authentication
- authorization
- parental consent
- retention และ deletion
- encryption in transit และ at rest
- audit log
- การแยกข้อมูลระหว่างครอบครัวหรือโรงเรียน

ห้ามถือว่าการออกแบบ IndexedDB ปัจจุบันสามารถนำไปใช้เป็นระบบบัญชีออนไลน์ได้โดยตรง

### 29. คะแนนฝั่ง client ไม่ใช่ข้อมูลที่เชื่อถือได้

XP และดาวใน IndexedDB สามารถแก้ไขได้จากเครื่องผู้ใช้ ซึ่งยอมรับได้สำหรับเกมออฟไลน์ที่ไม่มีผลภายนอก

หากมี leaderboard, ใบประกาศ, รางวัลจริง หรือรายงานผลให้โรงเรียน ต้องคำนวณและตรวจผลบน server และไม่เชื่อคะแนนจาก client โดยตรง

## Offline และ PWA

### 30. แก้ข้อความออฟไลน์ให้ตรงความจริง

ข้อความปัจจุบันระบุว่าข้อมูลบันทึกออฟไลน์ด้วย IndexedDB ซึ่งจริงสำหรับข้อมูลโปรไฟล์ แต่ตัวบทเรียน ฟอนต์ รูป และเสียงยังต้องโหลดจากเว็บ และยังไม่มี Service Worker สำหรับ offline app เต็มรูปแบบ

เลือกแนวทางใดแนวทางหนึ่ง:

**Online-first**

- เปลี่ยนข้อความเป็น “ความก้าวหน้าบันทึกไว้ในอุปกรณ์นี้”
- แสดงสถานะเมื่ออินเทอร์เน็ตหลุด
- มี retry ที่ชัดเจน

**Offline-capable PWA**

- เพิ่ม Web App Manifest
- เพิ่ม Service Worker
- cache app shell, JSON และ asset ที่จำเป็น
- แสดงความคืบหน้าการดาวน์โหลดบทเรียน
- กำหนด cache version และ update strategy
- ทดสอบ quota และการล้าง cache บน iOS/Android
- มี fallback เมื่อ asset บางรายการยังไม่ได้ดาวน์โหลด

## ข้อเสนอแนะด้าน Speech Scoring

ระบบ `similarity()` ปัจจุบันวัดการตรงกันของคำจาก transcript ไม่ได้วัดคุณภาพการออกเสียงจริง จึงไม่ควรเรียกว่า “คะแนนการออกเสียง” อย่างเด็ดขาด

ควรใช้ข้อความเช่น:

- “ระบบได้ยินคำตรงกับประโยคกี่คำ”
- “คะแนนความตรงของคำพูด”

หลีกเลี่ยงการทำให้เด็กคิดว่าสำเนียงของตน “ผิด” จากข้อจำกัดของ Speech Recognition

ควรให้ XP จากความพยายามและการฝึกซ้ำ ไม่ใช่เฉพาะ transcript ที่ระบบรู้จำได้ เพราะระบบอาจมี bias ต่อสำเนียงเด็กไทยและเสียงเด็ก

## Definition of Done ที่แนะนำ

ก่อนถือว่าเว็บพร้อมเผยแพร่ ควรผ่านรายการต่อไปนี้:

- ไม่มีข้อมูลผู้ใช้ถูกใส่ใน `innerHTML` โดยตรง
- ไม่มี inline event handler ที่ขัดขวาง CSP
- ใช้ HTTPS และมี security headers
- มี privacy notice ก่อนขอไมโครโฟน
- ใช้บทเรียนต่อได้แม้ไม่อนุญาตไมโครโฟน
- ซูมหน้าเว็บได้
- ใช้งานได้ด้วยคีย์บอร์ด
- โปรแกรมอ่านหน้าจอเข้าใจปุ่ม สถานะ และ feedback
- ไม่มี horizontal overflow ที่ viewport หลัก
- touch target สำคัญมีขนาดอย่างน้อยประมาณ 44 × 44 px
- ทดสอบ Chrome/Edge, Safari/iOS และ Android Chrome
- บันทึกและกู้คืนบทเรียนกลางคันได้
- มีวิธีลบโปรไฟล์และข้อมูลทั้งหมด
- ข้อความเรื่องออฟไลน์ตรงกับพฤติกรรมจริง
- ระบบคะแนนพูดไม่อ้างว่าเป็นการวัดสำเนียงอย่างแม่นยำ

## ลำดับดำเนินงานสำหรับ Claude

### Phase 1 — Security และ Privacy

1. แก้ XSS จาก profile และข้อมูล dynamic
2. เพิ่ม validation ชื่อเล่น
3. เพิ่ม privacy flow ของไมโครโฟน
4. จัดการ media stream และ Blob URL ให้ครบ
5. เอา `user-scalable=no` ออก
6. เตรียม CSP โดยลด inline handlers

### Phase 2 — Accessibility

1. เพิ่ม label และ aria-label
2. เพิ่มสถานะ tab, avatar และ progress bar
3. เพิ่ม live region
4. เพิ่ม focus-visible และ keyboard navigation
5. เพิ่ม reduced motion
6. ตรวจ contrast และ touch targets

### Phase 3 — Responsive UX

1. ปรับ grade selector บนมือถือ
2. เพิ่ม tablet/desktop layout
3. เพิ่ม save/resume กลางบท
4. เพิ่มเวลาโดยประมาณและจำนวนขั้น
5. เพิ่ม hint และ adaptive feedback

### Phase 4 — Deployment Hardening

1. Self-host fonts
2. เพิ่ม security headers
3. บังคับ HTTPS
4. ตรวจ MIME types และ caching
5. เลือก online-first หรือ PWA offline อย่างชัดเจน
6. ทดสอบอุปกรณ์และเบราว์เซอร์จริง

## ข้อควรระวังในการแก้ไข

- รักษาโครงสร้าง JSON และบทเรียนเดิมไว้ เว้นแต่จำเป็นต้อง migrate
- ทำ migration สำหรับข้อมูล IndexedDB เก่า หากเปลี่ยน schema
- อย่าลบความคืบหน้าของผู้ใช้ระหว่าง update
- แยก security refactor ออกจาก visual redesign เพื่อให้ตรวจ regression ได้ง่าย
- เพิ่ม automated tests สำหรับชื่อที่มีอักขระ HTML, navigation, save/resume และ permission denied
- ทดสอบกรณีไม่มีอินเทอร์เน็ต ไม่มีไมโครโฟน ปฏิเสธ permission ไม่มี Speech Recognition และ asset โหลดไม่สำเร็จ
